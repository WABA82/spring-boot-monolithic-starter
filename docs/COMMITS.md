# Commits Guide

> Conventional Commits을 기반으로한 커밋 컨벤션 문서입니다.
> 

---

## Conventional Commits 소개

커밋 메시지에 명확한 규칙을 부여하여 사람이 읽기 좋고, 기계가 자동화(버전 관리, 변경 로그 생성 등)하기 쉽게 만드는 표준 규약입니다. ([링크](https://www.conventionalcommits.org/ko/v1.0.0/))

## 메시지 규칙

### 기본 구조

커밋 메시지는 아래와 같은 구조를 가집니다.

```
<타입>[적용 범위(선택 사항)]: <설명>

[본문(선택 사항)]

[꼬리말(선택 사항)]
```

### **주요 커밋 타입 (Type)**

| **타입** | **설명** |
| --- | --- |
| **feat** | 새로운 기능 추가 |
| **fix** | 버그 수정 |
| **docs** | 문서 수정 (README 등) |
| **style** | 코드 의미에 영향을 주지 않는 변경 (포맷팅 등) |
| **refactor** | 버그 수정이나 기능 추가가 없는 코드 리팩토링 |
| **test** | 테스트 코드 추가 및 수정 |
| **chore** | 빌드 업무, 패키지 매니저 설정 등 기타 변경사항 |

### **주요 특징**

- **BREAKING CHANGE**: API의 호환성을 깨뜨리는 중대한 변경이 있을 경우, 타입 뒤에 `!`를 붙이거나 꼬리말에 `BREAKING CHANGE:`를 작성하여 명시합니다.
- **자동화 연동**: 이 규칙을 따르면 [**Standard Version**](https://www.conventionalcommits.org/ko/v1.0.0/) 같은 도구를 통해 자동으로 **CHANGELOG** 파일을 생성하거나 [**Semantic Versioning**](https://semver.org/lang/ko/)에 따라 버전을 올릴 수 있습니다.
- **협업 효율**: 팀원들이 커밋 로그만 보고도 어떤 종류의 변경이 일어났는지 즉각 파악할 수 있어 생산성이 높아집니다.

## 예제

### **기본 예제 (Type: Description)**

가장 많이 쓰이는 한 줄 커밋 형태입니다.

- **신규 기능:** `feat: 로그인을 위한 카카오 SDK 연동`
- **버그 수정:** `fix: 메인 페이지 이미지 로딩 지연 현상 수정`
- **문서 수정:** `docs: README 설치 가이드 최신화`

### **범위(Scope)를 포함한 예제**

수정된 **모듈이나 영역**을 괄호 안에 명시하여 맥락을 구체화합니다.

- `feat(auth): JWT 토큰 만료 처리 로직 추가`
- `fix(api): 검색 결과가 없을 때 404 에러 발생하는 문제 수정`
- `refactor(ui): 공통 버튼 컴포넌트 스타일 추출`

### **중대한 변경(Breaking Change) 예제**

이전 버전과 **호환되지 않는 큰 변화**가 있을 때 사용하며, 유의적 버전의 **MAJOR**를 올리는 근거가 됩니다.

```
feat: API 요청 구조 변경

본문: 기존 파라미터 방식에서 JSON Body 방식으로 변경되었습니다.

BREAKING CHANGE: 이제 `userId` 대신 `userGuid`를 필수값으로 받습니다.
```

### **본문과 꼬리말이 포함된 상세 예제**

변경 이유나 관련 이슈 번호를 남길 때 사용합니다.

**text**

```
fix(payment): 결제 시 간헐적으로 발생하는 타임아웃 오류 수정

- 네트워크 지연 시 재시도 로직을 3회 추가했습니다.
- 타임아웃 기본값을 5초에서 10초로 연장했습니다.

Fixes: #123, #125
```
